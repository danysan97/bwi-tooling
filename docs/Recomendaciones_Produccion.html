1 Hosting y Despliegue

AspectoEstado actualRecomendación
Hosting frontendLocal (npm run dev)Vercel o Netlify (gratis). Conectar el repo de GitHub, cada push a main despliega automáticamente.
DominioIP y puerto localComprar dominio (ej. tooling.bwi.com) y apuntarlo al hosting. Actualmente no es profesional ni seguro.
HTTPSSolo Supabase tiene SSLObligatorio. Sin HTTPS los navegadores muestran advertencias y los datos viajan en texto plano.
Deploy automáticoManual (git push)Configurar deploy automático en Vercel/Netlify al hacer push a main.

2 Variables de Entorno

AspectoEstado actualRecomendación
Credenciales SupabaseHardcodeadas en supabase.jsMover a archivo .env que nunca se suba a GitHub.
.env.exampleNo existeCrear con las variables sin valores para que otros desarrolladores sepan qué configurar.
.gitignoreVerificar que incluya .envAsegurar que .env esté en .gitignore.

3 Seguridad

AspectoEstado actualRecomendación
RLS (Row Level Security)Parcial (registro_horas tuvo problemas)Revisar que todas las tablas tengan RLS activo en Supabase: ordenes_trabajo, asignaciones_tecnicos, historial_orden, usuarios, materiales.
Rate LimitingNo existeImplementar límite de peticiones para prevenir flooding de comentarios o registros de horas.
Validación server-sideSolo frontendReforzar con triggers o funciones en PostgreSQL para que la BD rechace datos inválidos aunque el frontend falle.
AutenticaciónPIN (técnicos) / Password (admin)El PIN de 4-6 dígitos es débil para producción. Considerar auth con Supabase Auth + JWT.
⚠️ Crítico: Las credenciales de Supabase hardcodeadas en el código fuente son un riesgo de seguridad. Si el repo es público (o alguien accede al código), puede manipular la BD directamente.

4 Rendimiento

AspectoEstado actualRecomendación

Consultas a BDvista_ordenes carga todoCon tablas grandes (500+ órdenes) se volverá lento. Implementar paginación.

5 Experiencia Móvil / Taller

AspectoEstado actualRecomendación
Responsive designNo optimizado para móvilLos técnicos usan celulares en el piso de producción. Probar en pantallas pequeñas: touch targets, scroll, teclado virtual.
PWANo existeImplementar Progressive Web App para que los técnicos instalen la "app" en su celular y funcione con señal débil.
Touch targetsBotones pequeños en algunos lugaresLos botones y campos deben ser de al menos 44x44px para uso con dedos.
Modo offlineNo existeConsiderar Service Workers para cachear datos críticos cuando no hay internet.
💡 Dato clave: Los técnicos son los usuarios que más usarán el sistema diariamente (registros de horas, avances). Si la experiencia en celular es mala, no lo usarán.

6 Monitoreo y Errores

AspectoEstado actualRecomendación
Error trackingNo existeSentry (plan gratis): captura errores de JavaScript en producción y envía alertas por email/Slack.
Logs de BDNo se revisanRevisar periódicamente los logs de Supabase para detectar queries lentas o fallos recurrentes.
Métricas de usoNo existenConsiderar analytics básicos (Google Analytics o similar) para saber qué funciones se usan más.
Alertas de uptimeNo existenConfigurar un servicio de monitoreo (UptimeRobot, gratuito) que avise si el sitio cae.

AspectoEstado actualRecomendación

8 Testing

12 Escalabilidad de la Base de Datos

AspectoEstado actualRecomendación
vista_ordenesJOINs pesados en cada consultaCon crecimiento, denormalizar: guardar solicitante_nombre directamente en ordenes_trabajo en vez de calcularlo con JOIN.
historial_ordenCrecimiento continuoArchivar registros viejos (>1 año) en una tabla de respaldo para mantener consultas rápidas.
ÍndicesAlgunos creadosRevisar queries frecuentes con EXPLAIN ANALYZE y agregar índices donde falten.
ConexionesSupabase Free: 60 concurrentesSi muchos usuarios entran al mismo tiempo, considerar Supabase Pro (más conexiones).

★ Resumen de Prioridad

Clasificación de cada recomendación por nivel de impacto y urgencia:

🔴 Prioridad Alta — Hacer antes de producción

RecomendaciónRazón
Hosting + dominio + HTTPSSin esto no es usable en producción de forma segura y profesional.
Variables de entorno (.env)Las credenciales hardcodeadas son un riesgo de seguridad crítico.
Revisar RLS en todas las tablasProtección de datos. Sin RLS, cualquier usuario podría acceder a datos de otros.

🟡 Prioridad Media — Hacer en las primeras semanas

RecomendaciónRazón
Móvil responsive + PWALos técnicos usan celulares. Si no funciona bien, no lo adoptarán.
Sentry para erroresDetectar problemas automáticamente sin esperar reportes de usuarios.
Lazy loading del bundleMejorar tiempo de carga de 1.4 MB a ~300-400 KB iniciales.
GitHub Actions (CI)Prevenir errores de build antes de llegar a producción.

🟢 Prioridad Baja — Hacer con calma

RecomendaciónRazón
Tests automatizadosPrevención de regresiones a largo plazo.
NotificacionesMejora la experiencia pero no es urgente.
Documentación técnicaFacilita el mantenimiento y incorporación de nuevos desarrolladores.
Backups paid planDepende del volumen de datos y tolerancia a pérdida.
Escalabilidad de BDRelevante cuando la BD crezca significativamente.
